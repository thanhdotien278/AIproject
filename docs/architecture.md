# Architecture

## Runtime Shape

`backend/server.js` creates an Express app, wraps it in an HTTP server, attaches Socket.IO, configures EJS layouts, serves static files, mounts routes, connects to MongoDB, and listens on `0.0.0.0`.

```text
Browser
  |-- EJS pages: /, /register, /thankyou, /qrcode, /admin/*
  |-- React dashboard: /dashboard
  |-- JSON APIs: /api/*, /admin/api/*
  |-- Socket.IO: statsUpdated
Express server
  |-- routes/controllers
  |-- middleware
  |-- Mongoose models
MongoDB
```

## Main Layers

- **HTTP shell:** `backend/server.js`
- **Routes:** `backend/routes/register.js`, `backend/routes/admin.js`, `backend/routes/api.js`, `backend/routes/attendanceQr.js`, `backend/routes/speaker.js`
- **Controllers:** `backend/controllers/registerController.js`, `backend/controllers/adminController.js`, `backend/controllers/speakerController.js`
- **Middleware/services:** `backend/middleware/registrationValidation.js`, `backend/middleware/ipFilter.js`, `backend/middleware/attendanceQr.js`, `backend/services/attendanceQrStore.js`, `backend/services/totp.js`
- **Data:** `backend/models/*.js`
- **Views/assets:** `frontend/views/`, `frontend/public/`
- **Realtime React dashboard:** `frontend/components/RealTimeConferenceAnalyticsDashboard.jsx`, `frontend/react-dashboard/src/App.jsx`

## Realtime Design

The registration controller emits `statsUpdated` twice after successful registration: once for all conferences and once for the specific conference code. The React dashboard and admin dashboard both listen for `statsUpdated` and update only when the payload conference code matches the currently selected filter.

The event payload includes aggregate counts:

- `totalParticipants`
- `lunchCount`
- `dinnerCount`
- `transportCount`
- `hocVienCount`
- `donViNgoaiCount`

## Security Model

- Admin login checks a stored bcrypt password through `User.comparePassword`.
- Authenticated admin routes use `adminController.isAuthenticated`.
- There is no CSRF protection visible in the current routes.
- The reusable IP filter supports exact IPs and IPv4 CIDR ranges, but it is not mounted on registration.
- Dynamic QR tokens are random server-side values stored in process memory.

## Deployment Notes

- `railway.json` builds CSS with `npm run build` and starts with `npm start`.
- `ecosystem.config.js` configures PM2 cluster mode with 2 instances.
- Cluster mode is risky for dynamic QR because `attendanceQrStore` is process-local. A QR generated by worker A may be validated by worker B and fail as unknown.
