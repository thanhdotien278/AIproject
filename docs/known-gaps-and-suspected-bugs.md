# Known Gaps and Suspected Bugs

## High Impact

1. **`npm run seed` is broken by a syntax error.**
   - Evidence: `backend/config/seedAdmin.js` contains `eate new admin user` as bare text near line 35.
   - Verification: `node --check backend/config/seedAdmin.js` fails with `SyntaxError: Unexpected token 'new'`.
   - Impact: admin seed command can fail before creating the first admin account.

2. **Participant IDs are intended per-conference but enforced globally unique.**
   - Evidence: `Counter` uses per-conference keys like `participants_CONF`, but `Participant.participantId` has `unique: true`.
   - Impact: second conference participant `0001` can conflict with first conference participant `0001`.
   - Likely fix: compound unique index on `{ conferenceCode: 1, participantId: 1 }` and remove global unique participantId index.

3. **Dynamic QR does not actually check in a participant.**
   - Evidence: `/qr/checkin` validates a token, then redirects to `/register`.
   - Impact: no attendance state, check-in timestamp, check-in log, or participant identity is recorded.
   - Related report expectation: QR check-in should record presence and update the dashboard in realtime.

4. **IP filtering exists but is disabled for registration.**
   - Evidence: `backend/routes/register.js` comments out the IP filter import.
   - Impact: the network-based verification described in the QR report is not active.

5. **PM2 cluster mode conflicts with in-memory QR tokens.**
   - Evidence: `ecosystem.config.js` uses `instances: 2` and `exec_mode: "cluster"` while `attendanceQrStore` is a process-local `Map`.
   - Impact: QR generated by one worker can be validated by another worker and fail.
   - Likely fix: use a shared store such as MongoDB/Redis or disable cluster mode for QR-dependent deployments.

## Route and UI Mismatches

6. **Admin dashboard attendance buttons call a missing route.**
   - Frontend call: `PATCH /admin/api/participants/:participantId/attendance`
   - Registered routes: no matching route in `backend/routes/admin.js`.
   - Model mismatch: `Participant` has `attendance` boolean, but the dashboard uses `attendanceStatus`.

7. **Admin dashboard conference-specific bulk email route is missing.**
   - Frontend call: `POST /admin/api/conferences/:conferenceCode/send-bulk-emails`
   - Registered route: `POST /admin/send-bulk-emails`
   - Existing backend bulk email does not filter by selected conference.

8. **Route parameter conference registration paths are likely ineffective.**
   - Routes support `/register/:conferenceCode`, but `showRegisterForm` reads only `req.query.code`.
   - `registerParticipant` reads `req.body.conferenceCode`, not `req.params.conferenceCode`.

9. **Admin dashboard client script appears to contain duplicated/corrupted fetch code.**
   - Evidence: duplicate `fetchDashboardData` fetch block appears around `frontend/views/admin/dashboard.ejs`.
   - Impact: possible client-side syntax/runtime failure. This should be verified by rendering or parsing the page.

## Security and Operational Risks

10. **Hardcoded MongoDB Atlas fallback in seed script.**
    - Evidence: `backend/config/seedAdmin.js` includes a real-looking MongoDB Atlas URI fallback.
    - Impact: accidental writes to unintended remote DB if `.env` is absent.

11. **Server startup requires `MONGODB_URI`, despite README fallback examples.**
    - Evidence: `backend/server.js` calls `mongoose.connect(process.env.MONGODB_URI)` and comments out the local fallback.
    - Impact: local startup fails unless `.env` is configured.

12. **Admin authorization is authentication-only.**
    - Evidence: `User.userRole` exists, but `isAuthenticated` only checks session boolean.
    - Impact: manager/staff/receptionist roles are not enforced at the route layer.

13. **No visible CSRF protection for admin mutations.**
    - Impact: session-authenticated admin POST/PUT/DELETE endpoints may be exposed to CSRF if deployed without other controls.

14. **`x-forwarded-for` handling can be spoofed if the app is exposed directly.**
    - Evidence: IP extraction trusts the first `x-forwarded-for` value.
    - Impact: if IP filtering is enabled without a trusted reverse proxy boundary, clients can spoof the header.

15. **Public realtime dashboard exposes aggregate stats.**
    - Evidence: `/dashboard` and `/api/stats` are public.
    - Impact: acceptable for public reporting, but not for restricted operational dashboards.

## Implementation Drift From External QR Report

The QR report describes:

- dynamic QR token/JWT style payloads
- IP/Subnet verification
- attendance/check-in logging
- realtime dashboard updates from check-ins
- check-in performance targets

Current repo state:

- dynamic QR tokens are random in-memory strings, not JWTs
- TOTP helper exists and is tested but not wired into QR routes
- IP filtering exists and is tested but not mounted
- check-in redirects to registration and does not update attendance
- realtime dashboard updates only on registration-created stats
