- (constraint) Existing stack is Express CommonJS with EJS views and MongoDB/Mongoose; route/controller/model split should be preserved.
- (constraint) Current thankyou.ejs displays registration id only from a standalone participantId local, but /thankyou does not pass participantId; participant.participantId exists on the saved Participant/session data.
- (constraint) Current /thankyou reloads participant by email only and loads the latest conference by createdAt, so it can render data for a different conference or participant when emails/conferences overlap.
- (constraint) Current registration POST creates a Participant, stores participantName, participantEmail, conferenceName, conferenceCode, participantId, and participantData in session, then redirects the browser to /thankyou.
- (direction) Prefer thank-you rendering from the persisted Participant selected by stable registration identity, with session only carrying the lookup key after POST /register.
- (question) User should confirm which thank-you values are observed as incorrect; likely candidates from code are conference details, registration fields, participant identity when email appears in multiple conferences, and missing displayed registration id.
- (decision) Conference details shown on /thankyou must come from the Participant's conferenceCode, not from the latest Conference document. Binds event details and registrationFields on thankyou.ejs. Prevents mixing one participant with another conference's field list, dates, location, or contact information. Rule: /thankyou resolves Conference by participant.conferenceCode or the session conferenceCode fallback only when no persisted participant can be loaded.
- (decision) Thank-you page identity must be keyed by the participant created by the immediately preceding registration, not by email alone or latest conference. Binds registration POST and /thankyou. Prevents rendering another conference or another participant sharing the same email. Rule: POST /register stores a stable participant lookup key in session; /thankyou queries Participant by that key plus conferenceCode where applicable.
- (event) User confirmed targetAudience is incorrect on thank-you page; implementation stores participantMongoId after registration and resolves thank-you data from persisted participant plus participant conference.
